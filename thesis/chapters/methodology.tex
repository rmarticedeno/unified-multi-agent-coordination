\chapter{Research Methodology}
\label{chap:methodology}

\section{Research Paradigm and Study Type}

This thesis adopts a design-science paradigm because its central research output is an artifact: a coordination framework with explicit constructs, decision rules, an architectural instantiation, and an evaluation method. The study combines exploratory and analytical work in the literature review, formal and architectural design in the construction phase, and quantitative and qualitative analysis over preserved prototype traces. Quantitative measures assess classification, task execution, latency, and overhead; qualitative trace analysis examines whether authorization and refusal decisions are intelligible and supported by recorded evidence.

\section{Research Process}

The research process contains five linked stages. First, a critical literature review identifies the strengths and unresolved limitations of classical multi-agent coordination, symbolic reasoning, LLM agents, and interoperability protocols. Second, the research gap is translated into requirements for openness, interoperability, bounded authority, traceability, and explicit refusal. Third, these requirements are formalized as a protocol-independent framework. Fourth, the framework is mapped to a Python \texttt{CoordinationAgent} design over SDK middleware with protocol-specific communication handled as an implementation adapter. Fifth, controlled scenarios, Docker system checks, and local LLM reference batches evaluate whether the implementation preserves the proposed boundary. Each stage produces evidence for one or more specific objectives rather than serving as an independent implementation activity.

\section{Unit of Analysis and Scenario Design}

The primary unit of analysis is a coordination attempt: one problem request, one admitted registry snapshot, the candidate plan or refusal produced for that request, and the resulting execution trace when a plan is authorized. Scenarios are constructed by varying four factors: capability coverage, artifact compatibility, authority constraints, and runtime availability. This factorial view prevents evaluation from relying only on successful demonstrations.

The scenario suite must contain feasible direct-delegation cases, feasible decomposed cases, cases requiring an approved bounded linguistic transformation, and infeasible cases caused by missing capability, incompatible modality, cyclic dependency, insufficient authority, or unavailable agents. Each scenario includes a reference feasibility label and the minimum justification required for that label. Reference labels should be reviewed independently before system outputs are examined.

The executable evaluation is organized into three evidence levels. Unit and integration tests verify individual predicates, SDK adapters, service endpoints, retry behavior, and ledger recovery. Deterministic local end-to-end scenarios verify the coordination workflow without external services. Docker system scenarios then instantiate the coordinator, registry, and multiple independently deployed A2A fixture agents on a virtual network, so that registry discovery, A2A dispatch, artifact normalization, runtime failure, timeout, and trace reconstruction are exercised across process and container boundaries.

\section{Baselines and Ablations}

The implemented configuration is the \emph{hybrid CoordinationAgent}: it uses \texttt{CoordinationSdk} as the only communication boundary, may consume structured linguistic proposals, and always applies symbolic feasibility checks before dispatch. The present evaluation also records local LLM reference batches to characterize linguistic interpretation and candidate-plan behavior. These batches do not replace the coordinator and are not allowed to authorize execution.

Two comparative configurations remain useful for a larger follow-up study. A \emph{CoordinationAgent without symbolic feasibility} would receive the same request and registry descriptions but accept or reject its own proposed plan without the deterministic feasibility layer. A \emph{rule-only CoordinationAgent} would receive a structured request and apply the same explicit predicates without LLM interpretation. These variants would isolate the contribution of symbolic authorization and linguistic mediation, respectively. The current thesis therefore reports implemented prototype evidence and local LLM reference evidence, while avoiding unsupported claims of broad baseline superiority.

Additional ablations can remove one mechanism at a time: contract validation, explicit authority checks, trace recording, or bounded-auxiliary rules. These ablations should be evaluated only where the removed mechanism is relevant to the scenario. They are not treated as independent general-purpose systems.

Outcome metrics and process metrics are reported separately. A correct final artifact does not erase an unauthorized action or an invalid intermediate plan. Conversely, a justified refusal is a correct outcome for an infeasible request and must not be counted as task failure.

For the planned crash-fault-tolerant extension, robustness evaluation should add acceptance scenarios rather than empirical claims in advance. These scenarios should cover coordinator recovery from a durable ledger, transient registry outage, bounded task retry after timeout, alternate-agent replanning after an unavailable executor, and duplicate dispatch handling through idempotent identifiers. A run should be counted as crash-resilient only if the recovered trace reconstructs the same authorized session state and does not duplicate completed work. These scenarios must not be reported as completed or measured until the ledger, retry, and standby mechanisms have been implemented.

\section{Experimental Procedure}

For each scenario, every applicable configuration receives semantically equivalent inputs and an identical admitted-agent snapshot. Each run stores the interpreted request, candidate plan, predicate outcomes, authorization decision, coordination-agent decisions, SDK method calls, agent kind, transport or runtime path, messages, status transitions, artifacts, validation results, latency, and resource use. Aggregate results are accompanied by per-category error analysis. The implementation version, coordination-agent version, SDK version, protocol version, hardware, and software environment are fixed by the repository state and preserved reports used for the run.

The local LLM reference phase uses only models served locally at \url{http://127.0.0.1:1234}. The allowed model identifiers are \texttt{qwen/qwen3-1.7b} and \texttt{google/gemma-4-e2b}. No cloud LLM provider is used for submitted evidence. Because switching local models is expensive, the reference phase is batched: all checks are run for \texttt{qwen/qwen3-1.7b}, the model is switched once, and then all checks are run for \texttt{google/gemma-4-e2b}. The runner aborts if the endpoint does not advertise the requested model or if the completion response does not identify the requested model. Each batch records endpoint, model ID, prompt version, decoding settings, token usage, latency, and a model-specific run identifier.

For the Docker system phase, the simple operating model is one coordinator service, one registry service that publishes Agent Card URLs, many deterministic A2A fixture-agent containers, and one HTTP-only system-test runner. The runner must not inject Python objects into the coordinator. It calls the public service endpoints over the Docker network and writes a raw JSON report under \texttt{demo\_runs/}. No external LLM provider, vendor cloud agent, Kubernetes deployment, service mesh, or database is required for this phase.

\section{Analysis Strategy}

Feasibility decisions are compared with reference labels through a confusion matrix and category-level error rates. Paired scenario results are used for comparisons because configurations face the same requests. Descriptive statistics include central tendency and dispersion rather than averages alone. Where the sample size and distribution justify inference, paired statistical tests and effect sizes are reported. Qualitative analysis classifies false acceptance, false refusal, invalid decomposition, unsupported auxiliary synthesis, authority violation, artifact mismatch, and incomplete evidence.

\section{Validity and Reproducibility}

Construct validity depends on whether the metrics capture justified coordination rather than surface task completion. This threat is mitigated by separating plan validity, execution success, communication-boundary compliance, coordination-ownership compliance, and trace completeness. Internal validity may be affected by unequal prompts, model nondeterminism, or baseline implementation quality; shared inputs, repeated runs, versioned configurations, and paired comparisons reduce these effects. External validity is limited by the selected scenarios, admitted agents, SDK adapters, and protocol-specific implementation choices. The theoretical framework is protocol-independent, but empirical findings may not generalize to decentralized authorization, physical robots, adversarial registries, or other protocols. Conclusion validity depends on scenario count and variance and must therefore be assessed after data collection.

Reproducibility requires publication of scenario specifications, reference labels, registry snapshots, configuration files, prompts, raw traces, analysis scripts, exact dependency versions, and the service deployment configuration used for a run. Docker configuration can help fix the software environment for later replication, but it does not replace the preserved traces and scenario data required for empirical claims. Sensitive credentials and private agent endpoints are excluded, but their interfaces and response fixtures should be represented by reproducible substitutes.
