\chapter{Evaluación y resultados}
\label{chap:results}

\section{Estado de la evaluación}

At the present stage, the repository specifies the theoretical framework, implementation design, and evaluation protocol but does not contain a completed experimental dataset. Consequently, this chapter defines the exact reporting structure and acceptance criteria; it does not present fabricated measurements. Numerical tables must be populated only from preserved execution traces produced under the procedure in Chapter~\ref{chap:methodology}.

\section{Preguntas de evaluación}

The empirical evaluation addresses four questions. First, does the hybrid coordination agent distinguish feasible from infeasible requests more accurately than the baselines? Second, does symbolic authorization reduce false acceptance of plans with missing coverage, incompatible artifacts, cycles, or authority violations? Third, what latency, message, and model-use overhead does that validation introduce across remote A2A, local Python, and Lingo-backed agents? Fourth, do recorded traces permit an independent reviewer to reconstruct why a plan was authorized or refused, verify that agent invocation passed through \texttt{CoordinationSdk}, and verify that planning and feasibility remained owned by \texttt{CoordinationAgent}?

The last category is deliberately separated from planning infeasibility. A plan may be justified when authorized and still fail because the environment changes. Conflating these outcomes would obscure whether an error belongs to interpretation, planning, coordination, or execution.

\section{Reporte de resultados}

The primary results table must report, for each configuration, feasibility precision, feasibility recall, F1, invalid-plan rejection rate, task success rate, trace completeness, communication-boundary compliance, coordination-ownership compliance, median latency, message count, and model usage. Results must also be disaggregated by scenario category, agent kind, and runtime path. A single aggregate score is insufficient because a configuration can appear successful by accepting too many requests, by benefiting from a majority class, bypassing the SDK communication boundary, or placing planning authority in the wrong layer.

For every false acceptance and false refusal, the analysis records the first incorrect decision and its cause. Runtime failures are classified separately as agent unavailability, transport error, invalid artifact, timeout, or aggregation failure. The worked example in Chapter~\ref{chap:theoretical-solution} serves as an explanatory case, not as empirical evidence.

\section{Criterios para responder las preguntas de investigación}

PI1 se sustenta cuando las trazas de los escenarios muestran que los registros de capacidad, compatibilidad, dependencia, autoridad, contrato y completitud son suficientes para reconstruir la decisión de referencia. PI2 se sustenta cuando el \texttt{CoordinationAgent} híbrido conserva cobertura de interpretación y, a la vez, rechaza propuestas inválidas que la variante sin viabilidad simbólica acepta. PI3 se sustenta cuando las capacidades auxiliares se admiten únicamente bajo las reglas declaradas de elegibilidad y ciclo de vida, sin satisfacer experiencia sustantiva ausente. PI4 se responde mediante las mediciones comparativas de resultado, proceso, cumplimiento de frontera, cumplimiento de propiedad y sobrecarga. Hasta que esas mediciones se recopilen, estos elementos siguen siendo criterios de evaluación y no hallazgos empíricos.
