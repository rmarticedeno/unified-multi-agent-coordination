\chapter{Evaluation and Results}
\label{chap:results}

\section{Evaluation Status}

The evaluation executed three evidence layers. First, the non-Docker test suite verifies the core coordination records, feasibility predicates, SDK adapters, service endpoints, retry behavior, Lingo-facing coordinator surface, and ledger recovery. Second, deterministic local scenarios exercise the full coordination path without external services. Third, a Docker A2A system harness deploys one coordinator service, one registry service, multiple independently deployed A2A fixture agents, a coordinator configured with a failing registry, and an HTTP-only system-test runner. In addition, two local LLM reference batches were executed against \texttt{http://127.0.0.1:1234} using only \texttt{qwen/qwen3-1.7b} and \texttt{google/gemma-4-e2b}. These local LLM runs are not treated as authorizers; they are evidence about linguistic interpretation and candidate decomposition under the thesis boundary.

The preserved raw evidence is written under \texttt{demo\_runs/}. The generated analysis summary is written under \texttt{demo\_runs/thesis\_analysis/}. The local LLM batches use prompt version \texttt{local-llm-reference-v1}, temperature 0, schema-constrained JSON output, and one model per batch. The Qwen batch ran first, then the model was switched once and the Gemma batch was run.

Both local models produced parseable schema-valid JSON for every scenario and assigned candidate tasks only to registered agents. However, neither model reproduced the exact internal requirement names used by the symbolic fixtures. This is a useful negative result rather than a failure of the thesis: it shows why linguistic output must be mediated before it becomes an executable plan. The LLMs can produce structured candidate interpretations, but exact capability identity, artifact requirements, feasibility verdicts, and execution authorization remain the responsibility of the symbolic coordination layer.

\section{Research Question Interpretation}

RQ1 asks which coordination information must be represented explicitly. The deterministic and Docker scenarios show that capability names, registry entries, artifact contracts, dependency order, trust/authority predicates, validation contracts, and trace events are all needed to distinguish accepted plans, explicit refusals, and runtime failures.

RQ2 asks how responsibility should be divided between LLM reasoning and symbolic validation. The local LLM reference batches support the proposed division: the models can produce structured linguistic candidates, but exact requirement identity and verdict agreement remain imperfect. The coordination agent therefore treats model output as proposal evidence only and applies deterministic feasibility checks before dispatch.

RQ3 asks how narrow missing linguistic capabilities can be introduced without unbounded authority. The deterministic redaction scenario exercises the bounded auxiliary path: a narrow extraction step is admitted under explicit method, validator, and task-local lifecycle rules, while substantive missing capabilities in other scenarios are refused.

RQ4 asks which measures can determine whether the framework improves justified acceptance, explicit refusal, traceability, communication-boundary discipline, and coordination-ownership discipline. The current evidence demonstrates those measures on the implemented prototype: outcome agreement, refusal without dispatch, runtime-failure separation, Docker-level A2A boundary checks, model-specific local LLM traces, token usage, and latency are all recorded. The evidence does not establish broad benchmark superiority over all possible LLM-only or rule-only systems; such superiority would require a larger paired comparative study. It does establish that the implementation preserves the manuscript's main boundary: language proposes, symbolic coordination authorizes or refuses, and execution proceeds only through the SDK-managed communication surface.
