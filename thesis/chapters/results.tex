\chapter{Evaluation and Results}
\label{chap:results}

\section{Evaluation Status}

At the present stage, the repository specifies the theoretical framework, implementation design, service wrapper, Docker deployment configuration, and evaluation protocol. The implementation also includes unit tests, deterministic local scenarios, and a Docker real-network A2A system harness. Numerical claims in this chapter must still be populated only from preserved execution traces produced under the procedure in Chapter~\ref{chap:methodology}; passing infrastructure tests alone is not evidence of baseline superiority. Docker support improves reproducibility and supplies system-level traces, but comparative empirical conclusions require the baseline and ablation runs described below.

\section{Evaluation Questions}

The empirical evaluation addresses four questions. First, does the hybrid coordination agent distinguish feasible from infeasible requests more accurately than the baselines? Second, does symbolic authorization reduce false acceptance of plans with missing coverage, incompatible artifacts, cycles, or authority violations? Third, what latency, message, and model-use overhead does that validation introduce across remote A2A, local Python, and Lingo-backed agents? Fourth, do recorded traces permit an independent reviewer to reconstruct why a plan was authorized or refused, verify that agent invocation passed through \texttt{CoordinationSdk}, and verify that planning and feasibility remained owned by \texttt{CoordinationAgent}?

\section{Results Reporting}

The primary results table must report, for each configuration, feasibility precision, feasibility recall, F1, invalid-plan rejection rate, task success rate, trace completeness, communication-boundary compliance, coordination-ownership compliance, median latency, message count, and model usage. Results must also be disaggregated by scenario category, agent kind, and runtime path. A single aggregate score is insufficient because a configuration can appear successful by accepting too many requests, by benefiting from a majority class, bypassing the SDK communication boundary, or placing planning authority in the wrong layer.

For every false acceptance and false refusal, the analysis records the first incorrect decision and its cause. Runtime failures are classified separately as agent unavailability, transport error, invalid artifact, timeout, or aggregation failure. The worked example in Chapter~\ref{chap:theoretical-solution} serves as an explanatory case, not as empirical evidence.

\section{Criteria for Answering the Research Questions}

RQ1 is supported when the scenario traces show that capability, compatibility, dependency, authority, contract, and completion records are sufficient to reconstruct the reference decision. RQ2 is supported when the hybrid \texttt{CoordinationAgent} retains interpretation coverage while rejecting invalid proposals that the variant without symbolic feasibility accepts. RQ3 is supported when auxiliary capabilities are admitted only under the stated eligibility and lifecycle rules and do not satisfy substantive missing expertise. RQ4 is answered by the comparative outcome, process, boundary-compliance, ownership-compliance, and overhead measurements. Until those measurements are collected, these remain evaluation criteria rather than empirical findings.
